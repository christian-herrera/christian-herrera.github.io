Configurar el Terminal
...

Primero, habilitar el soporte SSH de GPG:

echo enable-ssh-support >> $HOME/.gnupg/gpg-agent.conf

Luego, usar el socket correspondiente, para esto hay dos versiones, una es directamente forzar el SSH_AUTH_SOCK:

export SSH_AUTH_SOCK="$(gpgconf --list-dirs agent-ssh-socket)"

Y la otra forma es dejar que el sistema cargue este valor cuando se inicie la terminal, para lograrlo, colocar las siguientes lineas en el archivo ~/.bashrc. Si utilizo zsh, entonces colocarlo en el archivo .zshrc

unset SSH_AGENT_PID
if [ "${gnupg_SSH_AUTH_SOCK_by:-0}" -ne $$ ]; then
  export SSH_AUTH_SOCK="$(gpgconf --list-dirs agent-ssh-socket)"
fi
export GPG_TTY=$(tty)
gpg-connect-agent updatestartuptty /bye >/dev/null

Luego, listar los llaveros disponibles, con los KeyGrip correspondientes:

gpg --list-keys --with-keygrip

Y cargar el keygrip de la clave que se desea usar:

echo 10F977F37B184A14DBF7533F5A7AC1E0903471D8 >> ~/.gnupg/sshcontrol

Para verificar que realmente funciona, usar:

ssh-add -l
ssh git@gitlab.com

Git Checkout
...

Permite volver a un Commit anterior, pero esto lo que hace es que el HEAD ahora apunte a un commit anterior y no al main. Aparecerá un mensaje indicando que el HEAD esta Desasociado. Esto significa que cuando vuelvas a camviar a una rama establecida, los Commits huérfanos estarán listos para que el recolector de basura los elimine .

ComandoDescripcion
git checkout a1e8fb5Hace que el HEAD apunte al commit indicado
git checkout mainVuelve al estado actual del proyecto

Git Clean
...

Permite eliminar los archivos que no estan con seguimiento (El uso mas comun).

ComandoDescripcion
git clean -fOpera sobre todos los archivos sin seguimiento, de manera forzada
git clean -fdIgual que el anterior, pero tambien elimina los directorios involucrados

Git Revert
...

Se le puede considerar un comando para deshacer, pero lo cierto es que no lo es en el sentido tradicional. En lugar de eliminar la confirmación del historial del proyecto, resuelve como invertir los cambios producidos por la confirmación y añade una nueva con el contenido inverso resultante.

ComandoDescripcion
git revert HEADRevierte el Estado del commit al que apunta HEAD
git revert <commit>Revierte el Estado del commit al que se especifica

Git Reset
...

A nivel superficial, git reset tiene un comportamiento parecido a git checkout. Mientras que git checkout solo opera en el puntero de referencia HEAD, git reset moverá el puntero de referencia HEAD y el puntero de referencia de la rama actual. Por ejemplo, teniendo el siguiente estado, donde la referencia HEAD apunta al ultimo commit d, y la referencia de la rama main tambien apunta al commit d.

Git Checkout b
Con git checkout b , la referencia main sigue apuntando a d. La referencia HEAD se mueve a b. Ahora el repositorio se encuentra en un estado de HEAD Desasociado.

Git Reset b
En comparación con el anterior, git reset mueve tanto las referencias de HEAD como las de la rama a la confirmación especificada.
La modificación del puntero de referencia sucede siempre y es una actualización del árbol de confirmaciones. Loas argumentos --soft, --mixed y --hard indican como modificar los arboles del índice del entorno de ensayo y del directorio de trabajo.

--hard
...

Es la opción mas directa, PELIGROSA y habitual. Cuando se utiliza, los punteros de referencia se mueven al commit especificado y al mismo tiempo, descarta todos los cambios no confirmados en el directorio de trabajo, es decir, todo lo que queda posterior al commit especificado.

ComandoDescripcion
git reset 67a4ca1 --hardVuelve al commit 67a4ca1 y deshace todo lo posterior

--soft
...

Cuando se pasa este argumento, se actualizan los punteros de referencia y el restablecimiento se detiene ahi, es decir, no descarta los cambios. Generalmente se utiliza para preparar los cambios realizados en commits anteriores para ser reconfirmados en un nuevo commit.

--mixed
...

Este es el modo de funcionamiento predeterminado y es el intermedio entre los dos anteriores. Afecta tanto al puntero de la rama (main) como al indice (area de preparacion), pero a diferencia de git reset --hard, no afecta al directorio de trabajo. Esto significa que los cambios en el directorio de trabajo no se descartan, pero los camios en el indice se restablecen al estado del commit especifico.

Git Rm
...

Este comando se utiliza para eliminar archivos de un repositorio de Git. Se puede considerar que es lo contrario a lo que hace git add. Un detalle es que gir rm no elimina ramas!.

ComandoDescripcion
git rm <Archivo>Quita el archivo del estado con Seguimiento.
git rm -r <Directorio>Quita el directorio del estado con Seguimiento
--cached

Al agregar al comando --cached, permite eliminar archivos del área de preparación (índice) pero conservarlos en el directorio de trabajo y en el historial de versiones.

Esto significa que los archivos ya rastreados y que habías agregado al área de preparación se eliminaran de esta área, pero se mantendran en tu directorio de trabajo y seguiran formando parte de los commits previos y futuros.
Para volver a incluirlos en el commit, se debera hacer un git add.

Git Remote
...

Permite crear, ver y eliminar conexiones con otros repositorios.
Por defecto, el repositorio remoto (por ejemplo: GitLab) se llama origin, mientras que el repositorio local tiene el nombre main.

Git Pull
...

Permite descargar todos los cambios del repositorio remoto en el repositorio local. Una vez descargado los cambios, automáticamente realiza un merge con la rama local para crear un commit fusionado con el nuevo contenido remoto.

ComandoDescripcion
git pullDe todos los repositorios remotos configurados
git pull <remote>Del repositorio remoto especifico, en estos casos, origin
git pull --no-commit <remote>En este caso, extrae el contenido remoto, pero no crea una nueva confirmacion de fusion.
git pull --rebase <remote>Mismo resultado que el anterior, pero ahora utiliza rebase.
git pull --verboseProporciona una salida detallada.

Git Fetch
...

Este comando descarga commit, archivos y referencias de un repositorio remoto a tu repositorio local, pero la diferencia con git pull es que no actualiza el estado de trabajo del repositorio local, por lo que tu trabajo actual no se vera afectado.

ComandoDescripcion
git fetch <remote>Recupera todas las ramas del repositorio remoto.
git fetch <remote> <branch>Recupera solo la rama especificada.
git fetch --allRecupera todos los repositorios remotos registrados y sus ramas.

Git Push
...

Se utiiza para cargar el contenido del repositorio local a un repositorio remoto.

ComandoDescripcion
git push <remote> <branch>Envía la rama especificada a una remota.
git push <remote> --forceEs igual que el anterior, pero fuerza el envio incluso si el resultado es una fusión sin avance rápido.
git push <remote> --allEnvia todas tus ramas locales a una rama remota especificada.

Git Merge
...

Me permite fusionar dos ramas especificas. Al momento de ejecutar este comando, la rama que se pase como argumento se fusionara con la rama en la que estoy posicionado, pero la rama que se trae, no se vera afectada.

Como primer paso, debo preparar la rama de recepción, para esto primero ejecuto un git status para asegurarme que la rama donde se realizara la fusión, se la correcta. Si debo moverme, ejecuto un git checkout.

Al momento de la Fusión, pueden existir diversos problemas de conflictos, por lo tanto, los casos mas comunes pueden ser:

Avance Rápido
...

Se da cuando el encabezado de la rama actual Target Branch es un antepasado de la rama que se pretende fusionar Source Branch.

Sin Avance Rápido
...

Si algún cambio en la rama de origen Source Branch entra en conflicto con la rama de destino Target Branch, Git le pedirá que resuelva el conflicto de la fusión.
Para el ejemplo, el ultimo commit en común con ambas ramas es el commit C. Pero en la rama origen, tiene dos commits posteriores, mientras que en la rama destino tiene uno solo.

En este caso, Git dejara en el archivo problemático, una leyenda donde debemos elegir con cual de las versiones quedarnos.

Fusión mediante Cambio de Base
...

Esta ultima opción es la que se vio con el comando Git Rebase, permite insertar los commits hechos en la rama Source Branch en la rama Target Branch de manera que la linea del tiempo se mantiene ordenada y creciente. Las modificaciones se realizan unicamente en la rama Target Branch.

Siguiendo el ejemplo del caso anterior, se puede ver que el orden correcto es el que llevan las letras en el abecedario, por tal motivo, al incertar los commits D y E en la rama Target Branch, el ultimo commit K ahora pasa a tener un nuevo predecesor que es E, por tal motivo, en la foto esta como K'.
